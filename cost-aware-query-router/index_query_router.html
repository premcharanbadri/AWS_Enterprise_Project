System 3: Cost-Aware Federated Query Router

1. What this actually does

Think of this system as a smart traffic cop for database queries. When our application needs data, it fires off a SQL query. Instead of hitting the database directly, it hits this router first.

The router looks at the query, figures out how heavy it is, and instantly sends it to the right place. Simple lookups go straight to Amazon RDS. Medium analytics go to AWS Athena. Massive, data-crunching joins get sent to Snowflake.

2. Why I built it

We have all seen what happens when a data analyst runs a massive `SELECT SUM` on a live production database. It locks the tables, spikes the CPU, and completely takes down the app for actual customers trying to use the product.

I built this to stop that from happening. It intercepts the traffic and automatically routes the heavy, analytical stuff away from our transactional databases.

3. Why C++17?

If you are routing thousands of queries a second, latency is everything. If I built this in Java or Python, the language's Garbage Collector would randomly pause the application to clean up memory. Even a 50-millisecond pause is unacceptable at scale.

I wrote this in bare-metal C++17 to get deterministic memory management. By using modern smart pointers (`std::unique_ptr`), memory cleans itself up instantly without background pauses, giving us predictable, microsecond response times.

4. How the routing logic works

I used the Strategy design pattern so the routing logic does not get tangled up with the actual database connection code.

To figure out where a query should go, the router doesn't actually run the SQL. It just reads the raw string, looking for structural keywords like `GROUP BY` or `JOIN`, and calculates a heuristic "weight" score. High score? It routes to Snowflake. Low score? It routes to RDS.

5. Real-world bugs I had to fix

Writing safe C++ is tricky. During the code review phase, I had to patch a few critical issues:

  • Regex compilation overhead: My regex was originally compiling from scratch on every single query, killing performance. I made the regex `static const` so it only compiles once at startup. Execution time dropped from microseconds to nanoseconds.
  • Memory corruption on ARM: Converting SQL text to uppercase was causing undefined behavior on ARM processors if the query had non-ASCII characters (like a Unicode table alias). I fixed this by explicitly casting the characters to `unsigned char`.
  • ORM spacing weirdness: ORMs automatically generate SQL, and they often put spaces in weird places (like `SUM (revenue)` instead of `SUM(revenue)`). My original router missed them, so I updated the regex to catch flexible whitespace.

6. The Bottom Line

This router solves a massive headache for the engineering team. It physically prevents long-running queries from crashing our customer-facing databases. It saves the company money by routing workloads to the most cost-efficient execution engine. Best of all, developers don't have to write complex routing logic into their apps—they just point their code at one URL, and the infrastructure handles the rest.
