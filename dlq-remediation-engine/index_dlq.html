System 1: Distributed DLQ Auto-Remediation System

1. What this actually does

Think of this system as a shock absorber for our microservices. When a downstream database goes offline, API requests start failing. Instead of losing that data or aggressively retrying, this system catches the failed messages, routes them to a Dead Letter Queue (DLQ), and carefully replays them when the database recovers.

2. Why I built it

When an outage happens, two issues usually follow. First is the "thundering herd": if 10,000 failed messages retry the exact second the database boots up, they instantly crash it again. Second is duplicate processing: network blips cause cloud queues to sometimes deliver the same message twice. If that message is a financial transaction, we just double-charged a customer.

I built this to stop both. It smooths out the retry traffic and mathematically guarantees we only process a message exactly once.

3. The system logic (Why I picked these tools)

Full Jitter vs. Standard Backoff

Standard exponential backoff tells a failing script to "wait 8 seconds." The problem is, if 10,000 messages all wait 8 seconds, they all hit the database simultaneously anyway. I implemented a "Full Jitter" algorithm to inject a randomized delay. It forces those messages to scatter their retries across a time window, physically flattening the concurrency spike so the database can breathe.

DynamoDB over Postgres for Idempotency

To stop duplicate transactions, I needed a distributed lock. A standard SQL database like Postgres would eventually bottleneck on connection pooling and table locks. I used DynamoDB Conditional Writes because it evaluates locks at the individual item level with single-digit millisecond latency. If a duplicate message shows up, the write fails instantly, and we safely drop the duplicate.

SQS Visibility Timeouts over time.sleep()

If a Python script needs to wait 10 seconds before trying again, the easy way is to write time.sleep(10). In AWS Lambda, you pay by the millisecond. Sleeping for 10 seconds burns paid compute time for absolutely no reason. Instead, I trigger the SQS Visibility Timeout API to literally hide the message on the queue for 10 seconds, and I kill the Lambda container immediately. The cost of waiting drops to zero.

4. Real-world bugs I had to fix

Getting this production-ready meant patching a few critical flaws during code review:

  • Boto3 Cold Starts: Initializing the AWS SDK inside the Lambda handler added massive latency during traffic spikes. I moved the initialization to the global scope so warm containers reuse the active connection pool.
  • Jitter Truncation: Small randomized jitter floats (like 0.4 seconds) were rounding down to zero in Python, causing instant retries instead of backing off. I fixed this by enforcing a mathematical floor of 1 second.
  • Database Memory Leaks: My idempotency locks were sitting in DynamoDB forever. I implemented a Time-To-Live (TTL) attribute so the locks automatically delete themselves after 24 hours.

5. The Bottom Line

This infrastructure acts as an insurance policy for the engineering team. Our downstream databases recover faster because they aren't hammered by synchronized retries. Financial and operational data stays pristine because duplicate messages are algorithmically rejected. And because it relies purely on event-driven serverless architecture, we don't pay a dime for idle polling servers.
